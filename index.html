Übung 2: SE Basics — Questions & Model Answers

Q1: Give 3 examples each of application software and system software. Are they clearly assignable?
Application software: Microsoft Word (word processing), Google Chrome (web browser), Spotify (music streaming). These serve end-user tasks.

System software: Windows 11 (operating system), Linux kernel, MySQL (database management system).

Not always clearly assignable: A web browser can be argued as both — it's application software for users but acts as system software for web apps. Similarly, a database can be system software or part of an application. The boundary is blurry because modern software stacks are layered.
Q2: Describe 5 characteristics of software. How does it differ from hardware?
5 software characteristics:
1. Intangibility — software has no physical form; you cannot touch it
2. No wear-out — software does not degrade with use (but requirements decay)
3. Complexity — modern software systems are extremely complex
4. Maintainability — software can be modified and extended after delivery
5. Reproducibility — software can be copied perfectly at near-zero cost

Hardware characteristics: Physical, wears out over time, manufacturing costs per unit, hardware failures are random events.

Key difference: Hardware degrades physically over time; software "degrades" only if environment changes (OS updates, new requirements). Hardware has fixed unit cost; software has near-zero marginal cost per copy.
Q3: Is the Deutsche Bahn app a Software, Software Product, or Software System?
It is a Software System (and also a Software Product).

Software: The code that runs the app.
Software Product: The app delivered to users with documentation, support, packaging.
Software System: The full ecosystem — app + backend servers + payment integration + real-time train data APIs + databases + external services.

It is Application Software because it serves a specific end-user task (travel management). It requires System Software (Android/iOS, web servers, network stack) to function.
Q5: Difference between Software Engineering and Programming?
Programming is the act of writing code to solve a specific problem. It focuses on implementation details.

Software Engineering is a systematic discipline that applies engineering principles to the entire software lifecycle — including requirements analysis, design, implementation, testing, deployment, and maintenance. It addresses large-scale, complex systems built by teams over time.

Analogy: A programmer is like a bricklayer; a software engineer is like an architect — they plan the structure, coordinate teams, ensure quality, and manage the full construction process.
Q6 & Q7: What is the Triangle of Tension (Spannungsdreieck) in software development?
The Triangle of Tension (also called "Iron Triangle" or "Magic Triangle") represents three competing constraints in software projects:

Corners: Scope / Quality / Time (Cost)
— Scope: Features and functionality to be delivered
— Time: Schedule and resources (budget)
— Quality: Reliability, correctness, performance

Why "tension": You cannot optimize all three simultaneously. Changing one forces trade-offs in the others. Adding more features (scope) requires more time OR reduces quality. Cutting time forces scope reduction OR quality suffers.

Example trade-off (Bahn-App): Prioritizing reliability (quality) → needs more testing time → reduces pace of new features (scope).
Q8: Two typical causes of software project failures?
1. Unclear or changing requirements — Projects fail when requirements are poorly defined, misunderstood, or change frequently without controlled management. Prevention: thorough requirements engineering, stakeholder workshops, change management process.

2. Underestimated complexity / poor planning — Teams underestimate the effort, leading to schedule overruns and budget exhaustion. Prevention: iterative development (Agile), regular effort estimation reviews, risk management planning.

Übung 3: Phase Models & SDLC

Q1: What are the key concepts of OOP? Difference between class and object?
4 core OOP concepts:
1. Encapsulation — hiding internal state, exposing only needed interfaces
2. Inheritance — child classes reuse/extend parent class behavior
3. Polymorphism — same method call behaves differently based on object type
4. Abstraction — modeling only relevant aspects, hiding complexity

Class vs. Object:
— A class is a blueprint/template (e.g., the concept of "Car")
— An object is a concrete instance of that class (e.g., "my red Toyota").

OOP vs. Procedural: Procedural code organizes logic as procedures/functions acting on data. OOP organizes data and behavior together in objects. OOP better models real-world entities.

Logic-based paradigms (e.g., Prolog) are based on formal logic and rules — the programmer declares facts and rules; the system infers conclusions.
Q2: The 5 key phases of Software Engineering and their tasks?
1. Requirements Analysis — Elicit, document, and validate stakeholder needs; produce requirements specification; identify functional and non-functional requirements.
2. Design (Architecture & Detailed Design) — Create system architecture, define components, design data models, create UML diagrams.
3. Implementation (Coding) — Write source code, conduct code reviews, unit testing during development.
4. Testing & Quality Assurance — Integration testing, system testing, acceptance testing, defect tracking and fixing.
5. Deployment & Maintenance — Release to production, monitor, fix bugs, adapt to new requirements.

Why phases blur in practice: Requirements evolve; design decisions emerge during coding; bugs found in testing require revisiting design. Real projects use iterative feedback loops.

UML by phase: Requirements → Use Case diagrams | Design → Class, Component, Sequence diagrams | Implementation → Code + Class diagrams | Testing → Activity diagrams for test flows
Q4: Rigid vs. Agile models — differences, examples, comparison table?
Rigid (sequential) models: Phases are strictly sequential; each phase must complete before next begins; changes are expensive late in the process.
Examples: Waterfall Model, V-Model

Agile models: Iterative, incremental delivery in short cycles (sprints); requirements can evolve; continuous customer feedback; working software delivered frequently.
Examples: Scrum, XP (Extreme Programming)

Q5: Why choose Scrum for a study-group app startup? Waterfall risks?
Choose Scrum because:
1. Requirements are unclear at start → Scrum allows discovering them iteratively
2. Dynamic market → Scrum's short sprints enable rapid adaptation
3. Early user feedback → prevents building the wrong product

Waterfall risks for this scenario:
— Requirements would be "frozen" but are actually unknown → massive rework
— Months of development before any user sees the product → late discovery of misunderstandings
— Changes are expensive → startup agility lost
Q6: Why is iterative development better than sequential for complex projects?
Iterative development allows teams to learn continuously and adapt throughout the project. Complex systems cannot be fully understood upfront — requirements evolve, technical challenges emerge, and user needs change. Each iteration delivers working software that reveals real feedback early. Sequential (waterfall) methods commit to a fixed plan that becomes outdated, making late changes extremely costly. Iterative approaches reduce risk by catching misunderstandings early, when corrections are cheap, rather than after months of wasted effort.

Übung 4: Stakeholder & Requirements Management

Q1: Define Functional vs. Non-Functional Requirements + Stakeholder + Quality Criteria
Functional Requirement: Describes what the system shall do — specific behavior or function.
Example (bike rental app): "The system shall allow users to reserve a bicycle by selecting a station and time slot."

Non-Functional Requirement: Describes how the system performs — quality attributes, constraints.
Example: "The reservation response time shall be under 2 seconds for 95% of requests."

Stakeholder: Any person or organization that has an interest in the system or is affected by it. Examples: End users (want easy booking), Admin (want analytics), Developer (want clean API).

4 Quality criteria for requirements:
1. Complete — all necessary information included
2. Unambiguous — only one possible interpretation
3. Testable — can be verified objectively
4. Consistent — no contradiction with other requirements
Q3: How to improve poorly written requirements?
"The app shall be fast." → Non-functional, but vague.
✓ Better: "The app shall display search results within 1.5 seconds for 99% of requests under normal load (≤1000 concurrent users)."

"Registration shall be simple." → Non-functional, vague.
✓ Better: "A new user shall be able to complete registration in ≤3 steps, requiring only email and password, with no more than 2 minutes elapsed."

"It shall be possible to reserve bicycles." → Functional, but incomplete.
✓ Better: "A registered user shall be able to reserve an available bicycle at a selected station for a chosen date and time slot. The system shall confirm the reservation within 3 seconds and send an email confirmation."

Test cases for reservation: (1) Valid reservation → confirmation shown + email sent; (2) Attempt to reserve already-booked slot → error message displayed
Q4: RFC 2119 prioritization (MUST/SHOULD/MAY) for bike rental app
MUST (mandatory): (a) User registration, (b) Bicycle reservation, (c) Credit card payment — these are core functionality without which the system fails to function.
SHOULD (highly recommended): (e) Push notifications, (d) Offline map mode — important for UX but system works without them.
MAY (optional): (g) Rating function, (h) Admin dashboard — nice-to-have features.
MUST NOT: Security violations (e.g., storing plaintext passwords).

Customer priority order (1=highest): (b) Reservation → (a) Registration → (c) Payment → (e) Push → (d) Offline → (f) GPS theft → (h) Dashboard → (g) Ratings
Q7: What is a Requirements Traceability Matrix (RTM)? Why important?
An RTM is a table that links each requirement to its design documents, code modules, and test cases. It ensures every requirement is implemented and tested, and makes the impact of changes visible.

Typical RTM columns: Requirement ID | Description | Design Document | Code Module(s) | Test Case ID(s) | Test Status

Benefits for testing: Ensures no requirement is untested; identifies which tests to re-run when a requirement changes; shows test coverage by requirement.

Risks without RTM: Requirements may be forgotten (not implemented); test coverage gaps; impact analysis of changes is impossible or error-prone.

If RQ-003 changes (add timestamp): RTM shows it affects DD-03 (design spec), TaskManager.java (code), and TC-03 (test case) — all must be updated.

Übung 6: Software Design & Architecture

Q1 Multiple Choice Answers (Architecture basics)
1. Software architecture (IEEE): (b) — fundamental organization of a system embodied in components, relationships, and principles.
2. Architecture vs. Design: (b) — Architecture is high-level (system-wide), Design is low-level (code-near).
3. False statement about C4: (c) — C4 is NOT a formal language like UML; it is a lightweight modeling approach/notation.
Q2: Grobentwurf vs. Feinentwurf. What is High/Low Cohesion and Low Coupling?
Grobentwurf (Coarse-grained design / System Architecture):
— High-level view of the system; identifies major components and their interactions
— Defines the overall architecture style, system boundaries, external interfaces
— Typical artifacts: C4 Context/Container diagrams, Component diagrams

Feinentwurf (Fine-grained design / System Design):
— Low-level details; class structures, method signatures, data types
— How individual components are internally structured
— Typical artifacts: UML Class diagrams, Sequence diagrams

High Cohesion: All elements within a module/class work toward a single, well-defined purpose. A class should do one thing well.
Low Coupling: Modules/classes should depend on each other as little as possible. Changes in one module should minimally affect others.
Importance for Grobentwurf: High cohesion makes components understandable and maintainable. Low coupling enables independent development, testing, and replacement of components.
Q3: C4 Model — all 4 levels explained
The C4 model uses 4 levels of abstraction to describe a software system:

Level 1 — Context (System Context Diagram): Shows the system in its environment — who uses it (persons/actors) and what external systems it interacts with. Audience: non-technical stakeholders.

Level 2 — Container: Zooms into the system; shows the deployable units (web app, mobile app, API, database). Each container runs separately. Shows technologies used. Audience: developers and architects.

Level 3 — Component: Zooms into a single container; shows the internal components/modules and their interactions. Audience: developers.

Level 4 — Code: Zooms into a single component; shows classes/functions. Usually auto-generated from code. Audience: individual developers.
Q4: Architecture patterns — which for a fitness app?
Recommended: Microservices Architecture
Arguments: (1) Independent scaling of services (e.g., scale only the Workout service during peak usage); (2) Different teams own different services (Social, Training, Wearables); (3) Wearables integration can be a separate Adapter service, easily replaceable; (4) Offline functionality can be handled by a dedicated sync service.

Alternative: Event-Driven Architecture — fits well for social sharing (events: WorkoutCompleted → notify friends) and wearable data streams.

Architecture Patterns overview (exam critical):
— Layered: UI→Logic→Data; simple, good for small systems
— Microservices: small independent services; scalable, complex ops
— Event-Driven: components communicate via events; loose coupling
— Hexagonal (Ports & Adapters): core logic isolated from I/O; testable
— Pipes & Filters: data flows through transformation steps; good for streaming/data processing
Q5: What are Design Patterns? GoF 3 categories with examples?
Design Patterns are reusable solutions to commonly occurring problems in software design. They are not code to copy but templates describing how to solve a problem in a specific context. Introduced by the "Gang of Four" (GoF) book (Gamma, Helm, Johnson, Vlissides).

3 GoF categories:
1. Creational patterns — how objects are created
Example: Singleton (only one instance of a class), Factory Method (delegate object creation to subclasses)

2. Structural patterns — how classes/objects are composed
Example: Adapter (makes incompatible interfaces work together), Decorator (add behavior without subclassing)

3. Behavioral patterns — how objects communicate
Example: Observer (notify dependents of state changes), Strategy (switch algorithms at runtime)

Übung 7: UML Diagrams

Q1: UML multiple choice correct answers
1. Correct: "UML serves for modeling and visualization of software systems" (not a programming language; works for non-OO systems too)
2. For interactions over time: Sequence diagram
3. Class diagrams show: structure of classes, attributes, and relationships
4. Structural (static) diagrams: Class diagram, Object diagram, Component diagram (Activity and Sequence are behavioral)
5. Use Case diagram elements: Actors, Use Cases, Relationships (include/extend) (not classes, not activity states)
Q2: What is a Context Diagram? Relation to C4 model?
A Context Diagram shows a system in relation to its environment — it identifies the system boundary and shows what external actors (users, other systems) interact with it. It does NOT show internal structure, implementation details, or data flows.

C4 relation: The C4 Model's Level 1 (System Context) IS a context diagram. In C4 notation, it shows: the system (central box), people (actors), and external software systems, with labeled relationships.

What is intentionally omitted: Internal components, database structure, technologies, message protocols, data formats. These appear in Level 2 (Container), Level 3 (Component), or detailed UML diagrams.
Q4: UML Relationships — Association, Aggregation, Composition, Inheritance
Association: A general "uses" or "knows about" relationship. → Represented by a plain line. Multiplicity shown at ends (e.g., 1..*).

Aggregation (weak "has-a"): A whole-part relationship where the part can exist independently of the whole. → Diamond (hollow) at the whole end. Example: Department has Employees (employees exist without the department).

Composition (strong "has-a"): A whole-part relationship where the part CANNOT exist without the whole. → Diamond (filled) at the whole end. Example: Faculty has a Library (library belongs exclusively to that faculty).

Inheritance (generalization): "is-a" relationship; child class extends parent. → Arrow with hollow triangular head pointing to parent. Example: Circle extends Shape.

Realization (interface implementation): Class implements an interface. → Dashed arrow with hollow triangular head.

Dependency: One class uses another temporarily. → Dashed arrow.
Q6: Sequence diagram — key UML elements
Key elements of a UML Sequence Diagram:
— Lifelines: Vertical dashed lines representing participants (actors, objects, systems)
— Activation bars: Rectangles on lifelines showing when a participant is active
— Messages: Horizontal arrows between lifelines (synchronous: solid arrow; async: open arrow; return: dashed arrow)
— Alt/Opt frames: Combined fragments for alternative flows (alt = if/else; opt = optional; loop = loops; par = parallel)

For the "Dozent uploads material" flow: Dozent → Website → Backend → MinIO/Storage (save file) → Database (save metadata) → response chain back

Alternative flow: If file type invalid → Backend returns error → Website shows error message to Dozent

Übung 8: SOLID Principles

All 5 SOLID Principles — full explanations
S — Single Responsibility Principle (SRP): A class should have only ONE reason to change — it should do one thing only. Example violation: OrderService that also handles logging and email sending. Fix: split into separate classes.

O — Open/Closed Principle (OCP): Software entities should be open for extension but closed for modification. Add new behavior via inheritance/composition, not by changing existing code. Example: add a new payment method by creating a new class implementing PaymentInterface, without touching existing payment code.

L — Liskov Substitution Principle (LSP): Objects of subclasses must be usable wherever objects of the parent class are expected, without breaking correctness. If Bird has fly(), then Penguin cannot extend Bird if Penguin cannot fly().

I — Interface Segregation Principle (ISP): Clients should not be forced to depend on interfaces they don't use. Split large interfaces into smaller, role-specific ones. Violation example: IUniversalService with pay(), calculateDiscount(), logAction() — forces classes to implement methods they don't need.

D — Dependency Inversion Principle (DIP): High-level modules should not depend on low-level modules — both should depend on abstractions. And abstractions should not depend on details. Use interfaces/abstract classes rather than concrete implementations. Enables dependency injection.
Q1 MC answers: SOLID multiple choice
1. "Only one reason to change" → Single Responsibility Principle
2. OCP correct statements: Open for extension, closed for modification AND Add via inheritance/composition
3. Subclasses replaceable wherever superclass expected → Liskov Substitution Principle
4. Best OrderService implementation → public Order createOrder(int customerId, int productId) { return new Order(1, customerId, productId); } (matches UML: returns Order, takes 2 params)
5. Interface statements: defines how to interact + serves decoupling + class can implement multiple interfaces (NOT: interfaces cannot contain implementation details — actually in Java 8+ they can have default methods, but traditionally FALSE)
7. Component interfaces: defines services offered + can serve exchangeability (NOT: always contain implementations)
Q3: Identify SOLID violations in the IUniversalService diagram
Violation 1 — ISP (Interface Segregation): IUniversalService combines pay(), calculateDiscount(), logAction() — three unrelated responsibilities. ShoppingCart is forced to depend on logAction() even if it only needs discounts.
Fix: Split into IPaymentService, IDiscountService, ILoggingService.

Violation 2 — SRP (Single Responsibility): ShoppingCart handles items AND payment processing (processPayment method) — two distinct responsibilities.
Fix: Remove processPayment from ShoppingCart; delegate to a separate CheckoutService.

Violation 3 — OCP (Open/Closed): PaymentService has PaypalPay() and CreditCardPay() as concrete methods. Adding a new payment type requires modifying PaymentService.
Fix: Create a PaymentStrategy interface with a pay() method; implement PaypalPayment, CreditCardPayment as separate classes.
Q5: Activity Diagram — Euclidean Algorithm
Euclidean Algorithm: gcd(a, b) — finds greatest common divisor

Activity Diagram structure:

↓
② Action: "Input a, b"
↓

Übung 9: Software Quality & Testing 1

Q1: MC answers — Testing concepts
1. Test pyramid layers: Unit Tests, Integration Tests, System Tests (not "Stakeholder Tests")
2. Correct testing statement: "Testing increases software quality" AND "Combination of automated and manual tests is usually sensible" (High coverage ≠ always sensible; 100% coverage ≠ bug-free)
3. V&V: "Beta tests with real users are an example of validation" AND "Load tests contribute to verification" (Verification ≠ user requirements fulfilled — that's validation)
4. Test planning: "Should start early" + "Risk-based testing prioritizes critical areas" + "Traceability links tests to requirements" (Planning does NOT guarantee bug-free)
5. Test pyramid: "70/20/10 is a heuristic" + "Ice Cone anti-pattern leads to slow pipelines" (Many UI tests is NOT best practice)
7. Version with many new features breaking existing requirements → neglected: Functional Suitability
9. App fails completely on small network errors → missing: Robustness
Q2: Verification vs. Validation — classify the scenarios
Verification ("Are we building the product right?" — conformance to specification):
(a) Check if specified fields are correctly implemented ✓
(c) Unit tests for input validation ✓
(e) Automated test checks database persistence as specified ✓
(f) Check Pflichtenheft layout rules ✓

Validation ("Are we building the right product?" — meeting real user needs):
(b) Real users test if app helps them manage appointments faster ✓
(d) Product Owner evaluates reminder function usability ✓
(g) Focus group evaluates if weekly view helps them ✓
Q3: Manual vs. Automated Testing — classify scenarios
Manual testing:
(a) Checking if checkout workflow is logically understandable → subjective, human judgment needed
(c) Exploratory testing for unexpected bugs → creativity required
(e) UX designer checks buttons, colors, icons → visual/aesthetic judgment

Automated testing:
(b) Every build checks all payment methods → regression test, must be fast and repeatable
(d) 1,000 shopping carts → load test, impossible manually at scale
(f) Nightly regression suite for login, logout, article display → repetitive, time-critical
Q4: Test levels — Unit → Integration → System → E2E
Unit Test: SUT = single method/class (e.g., CourseService.enrollUser()). Dependencies: all replaced by Mocks. Goal: logic bugs in single unit. Example: enrollUser() with valid ID → returns success

Integration Test: SUT = CourseService + its real database. External services (Email, other services) still mocked. Goal: interface/data layer bugs. Example: enroll user → check database row is created

System Test: SUT = full deployed system (Course + User + Notification services) against real requirements. UI not tested. Goal: functional requirement coverage. Example: "User can enroll in course" end-to-end via API

E2E Test: SUT = complete system from UI. All services real, near-production environment. Goal: business process validation. Example: Open browser → login → search course → enroll → receive confirmation email

Übung 10: Testing 2 — White-box, Black-box, TDD

Q1: White-Box Testing — CFG, Statement Coverage, Branch Coverage, Term Coverage
Calculator.getTotal() — Control Flow Graph (CFG):
Node 1: total=0, i=0 (initialization)
Node 2: while i < limit? (loop condition)
— NO → Node 5: return total
— YES → Node 3: if i%2==0 AND factor>1?
  — YES → total += i * factor
  — NO → total += i
Node 4: i++ → back to Node 2

Statement Coverage (all statements executed at least once):
TC1: Calculator(3, 2) — limit=3, factor=2 → i runs 0,1,2. i=0: even+factor>1 → total+=0*2; i=1: odd → total+=1; i=2: even+factor>1 → total+=4. Covers all statements including both branches of the if.

Branch Coverage (all edges traversed — need MINIMUM test cases):
TC1: Calculator(0, 2) → while condition FALSE → loop not entered (covers while=false branch)
TC2: Calculator(3, 2) → covers while=true, if(even AND factor>1)=true AND false (i=1 is odd)
→ 2 test cases cover 100% Branch Coverage

Term/Condition Coverage (each individual boolean term evaluated TRUE and FALSE):
Condition: (i%2==0) AND (factor>1) — two terms, each must be T and F independently.
TC-A: Calculator(4, 2) → i=0: i%2==0 is T, factor>1 is T
TC-B: Calculator(4, 1) → factor>1 is F (covers factor>1=FALSE)
TC-C: Calculator(3, 2) → i=1: i%2==0 is F (covers i%2==0=FALSE)
→ 3 test cases cover 100% Term Coverage
Q2: Black-Box Testing — Equivalence Classes and Boundary Values for ShippingService
Equivalence Classes for weight [0.0, 50.0]:
EC1 (invalid): weight < 0 → IllegalArgumentException
EC2 (valid, <1.0): 0.0 ≤ weight < 1.0 → base fee = 5€
EC3 (valid, 1–10): 1.0 ≤ weight < 10.0 → base fee = 10€
EC4 (valid, ≥10): 10.0 ≤ weight ≤ 50.0 → base fee = 20€
EC5 (invalid): weight > 50.0 → IllegalArgumentException

Equivalence Classes for distance [0, 2000]:
EC6 (invalid): distance < 0 → exception
EC7 (valid): 0 ≤ distance ≤ 2000 → +2€ per started 100km
EC8 (invalid): distance > 2000 → exception

Boundary Values for weight:
-0.01 (invalid) | 0.0 | 0.99 | 1.0 | 9.99 | 10.0 | 50.0 | 50.01 (invalid)

Cost example: weight=5.0 (EC3), distance=150 → 10€ + 2*2€ = 14€ (150/100 = 1.5, rounds up to 2 × 100km blocks)
Q3: TDD — what is it and how does ParkingFeeCalculator work?
TDD (Test-Driven Development): A development methodology where tests are written BEFORE the implementation code.

TDD cycle (Red → Green → Refactor):
1. Red: Write a failing test for the next small piece of functionality
2. Green: Write the minimum code to make the test pass
3. Refactor: Clean up code without changing behavior
Repeat for each new feature.

ParkingFeeCalculator test cases:
— lostTicket=true → 3000 cents (30€)
— duration=0, lostTicket=false → 0 cents (first 10 min free)
— duration=10, lostTicket=false → 0 cents (still free)
— duration=11, lostTicket=false → 200 cents (1st hour: 2€)
— duration=60, lostTicket=false → 200 cents (1 hour)
— duration=61, lostTicket=false → 400 cents (2nd hour started)
— duration=-1 → IllegalArgumentException
— lostTicket=true overrides duration → always 3000 cents

Übung 11: Software Operation & Maintenance

Q1: Eisberg Model, Bathtub Curve, Technical Debt, Ostrich Algorithm
Eisberg (Iceberg) Model: Like an iceberg, the visible part (initial development) is small; the majority of costs (60%+) are below the surface — maintenance, operations, monitoring, bug fixes, new features. Correct: "A large part of total costs arise after the initial release."

Bathtub Curve phases:
— Early failures (infant mortality): Häufung von Konfigurationsfehlern direkt nach Release
— Stable phase (normal operation): Seltene, schwer reproduzierbare Fehler nach langer Laufzeit
— Aging/Drift: Zunehmende Störungen durch veraltete Abhängigkeiten und technische Schulden

Ostrich Algorithm (ignore the problem):
(a) UI glitch at 0.01%, no payment impact → Acceptable to ignore (low risk, low impact)
(b) Wrong invoice amounts at 0.01% → NOT acceptable (financial/legal risk)
(c) Rare race condition with data loss → Borderline — needs risk assessment; data loss is severe, but rarity and fix cost must be weighed
Q2: Maintenance types — Lientz/Swanson classification
4 maintenance types:
— Corrective: Fix defects found after release
— Adaptive: Adapt to changed environment (new OS, new laws, API changes)
— Perfective: Improve quality, performance, usability — not fixing bugs
— Preventive: Proactive work to prevent future problems (refactoring, technical debt reduction)

Classification:
(a) DSGVO compliance changes → Adaptive
(b) API v2 → v3 migration → Adaptive
(c) Performance tuning p95 latency → Perfective
(d) NullPointerException after release → Corrective
(e) Refactoring + test coverage → Preventive
(f) Remove dead code/feature flags → Preventive

Why preventive maintenance is hard to sell: No visible output; business sees cost without immediate benefit; "if it ain't broke, don't fix it" mindset. Counter-arguments: reduces long-term costs; prevents outages; faster future development.
Q4: SLA calculation + Monitoring + MTTR
Max downtime at 99.95% availability:
Total minutes/year = 365 × 24 × 60 = 525,600 min
Allowed downtime = 525,600 × 0.0005 = 262.8 minutes ≈ 4.38 hours/year

Why additional "nines" are disproportionately expensive:
99.9% → 526 min downtime/year (relatively easy: good deployment practices)
99.99% → 52.6 min/year → requires redundant infrastructure, fast auto-failover, 24/7 on-call
Each additional nine cuts allowed downtime by 90%, requiring massive infrastructure investment.

MTTR (Mean Time To Repair/Recovery): Average time to restore service after an incident.
— Internal reporting service: MTTR goal ~4-8 hours (not customer-facing)
— Checkout service: MTTR goal <15 minutes (every minute = revenue loss)

Metrics: HTTP request rate, p95 response time, CPU usage

Q3: Legacy systems — strategies and Strangler Pattern
For the described legacy system (12 years old, no CI, high business value, EOL runtime):
Best strategy: Re-Engineering + Strangler Pattern (not replacing outright — too risky with high business value and poor documentation)

Strangler Fig Pattern: Gradually replace the legacy system by building new functionality around it. New features are built as separate services. Over time, functionality is "strangled" out of the monolith and into the new system. The old system handles remaining traffic until it can be decommissioned. Key benefit: zero-downtime migration; low risk; incremental delivery. Steps: (1) identify stable domains, (2) build new service for that domain, (3) redirect traffic via API Gateway, (4) retire old module, (5) repeat.

Q2 — Spannungsdreieck (4 pts)
Triangle of Tension
Drawing: An equilateral triangle with one corner each labeled: Scope (Features), Time/Cost, Quality

Explanation: The triangle of tension (Iron Triangle) represents the three fundamental constraints of any software project. Each corner represents a dimension: scope (what is built), time/cost (resources available), and quality (how well it works). These constraints are in tension because optimizing one negatively affects the others. For example, adding more features (scope) requires either more time/cost or reduced quality. This is why it is called a "tension" triangle — there is a constant pulling force between the three dimensions that project managers must balance.
Q3 — Grobentwurf vs. Feinentwurf (4 pts)
Coarse vs. Fine Design
Grobentwurf (Coarse/System Architecture): High-level design that defines the overall structure of the system — its major components, their responsibilities, and how they interact. It answers "what major parts exist and how do they connect?" UML: Component Diagram or C4 Container Diagram.

Feinentwurf (Fine/System Design): Low-level, detailed design that specifies internal structure of each component — class hierarchies, attributes, method signatures, algorithms. It answers "how is each component internally structured?" UML: Class Diagram or Sequence Diagram.
Q4 — Structure vs. Behavior Diagrams (2 pts)
Structural (static) diagrams describe the static aspects of the system — what elements exist and how they relate. Examples: Class Diagram, Component Diagram, Object Diagram, Package Diagram.

Behavioral (dynamic) diagrams describe how the system behaves over time — the flow of control and data, interactions, state changes. Examples: Sequence Diagram, Activity Diagram, State Machine Diagram, Use Case Diagram.
Q5+Q6 — S and O in SOLID (2+2 pts)
S — Single Responsibility Principle: A class should have exactly one reason to change — it should be responsible for exactly one piece of functionality. This makes classes easier to understand, test, and maintain.

O — Open/Closed Principle: Software entities (classes, modules) should be open for extension but closed for modification. New behavior is added by creating new classes or using inheritance/composition, not by changing existing code. This prevents regression bugs and keeps existing tests valid.

Q8 — Architecture Style for Smart City Traffic (6 pts)
1. Disadvantage of current approach (monolithic, single processing step): Tight coupling makes it impossible to test or change individual processing steps in isolation. A bug or change in one step risks breaking the entire pipeline.

2. Recommended style: Pipes & Filters Architecture
Argument 1: The system is fundamentally a data processing pipeline (sensor data → clean → aggregate → analyze → output), which maps perfectly to Pipes & Filters.
Argument 2: Each processing step (filter) can be developed, tested, and replaced independently, solving the current isolation problem.
Argument 3: New analysis steps can be inserted into the pipeline without modifying existing filters (OCP-compatible).

3. Disadvantage of Pipes & Filters: The slowest filter limits overall throughput (bottleneck). Error handling and retry logic across a pipeline is complex.

Q10 — Aktivitätsdiagramm: Online Shop (5 pts)
⚠ Draw on paper — must include all 4 required elements
Scenario: Customer adds items to cart → decides checkout or continue shopping → at checkout: payment + shipping run IN PARALLEL → when both done → order complete.

Required elements (stated in exam): Initial Node ● | Activity Final Node ◉ | Decision/Merge Node ◇ | Fork Node ■ (thick bar — most commonly forgotten!)

Diagram structure:
● Initial Node
↓
□ "Artikel in Warenkorb legen"
↓
◇ Decision: "Zur Kasse gehen?"
  — [Nein] → □ "Weitere Artikel hinzufügen" → loop back ↑

↓
◉ Activity Final Node

Key points: Fork and Join are both drawn as thick horizontal bars ■. Fork = one flow splits into many parallel flows. Join = parallel flows synchronize before continuing. The Decision node ◇ handles the loop (continue shopping), not the parallel split.
